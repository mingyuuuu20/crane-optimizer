# 🔍 타워크레인 최적화 — 전체 근거 감사 보고서

목적: 모든 상수·수식·제약이 **(A) 출처 있는 실제 기준인지 / (B) 임의로 만든 값인지** 분류.
원칙: "임의로 만든 값"이면 논문에서 그렇게 솔직히 밝혀야 하고, 가능하면 출처를 찾거나 민감도 분석으로 보완해야 함.

분류 기호:
- 🟢 **검증됨**: 실제 기준/출처 확인, 숫자도 합리적
- 🟡 **의심**: 출처는 그럴듯하나 숫자 미확인 또는 약간 안 맞음 → 확인·보정 필요
- 🔴 **임의/오류**: 근거 없이 임의로 정한 값, 또는 출처와 숫자가 명백히 안 맞음 → 반드시 수정

## 🎯 결론 및 권고

**현 상태: 뼈대(수식 구조, 일부 표준)는 근거 있으나, 핵심 "숫자"의 상당수가 출처 없는 임의값.**

이건 학부 캡스톤에선 흔한 일이지만, **A+/논문 게재를 노린다면 정직하게 처리해야 함:**

### 옵션 1 — 정직한 명시 (현실적, 추천)
임의값들을 논문에서 **"공학적 판단에 의한 가정(assumption)"으로 명시** + 가장 민감한 2~3개(P값, η, 가중치)에 **민감도 분석** 추가. "이 값을 ±50% 바꿔도 Pareto front 패턴은 유지된다"를 보이면 방어 가능.

### 옵션 2 — 출처 보강
각 임의값에 대해 실제 기준(제조사 카탈로그, KOSHA 원문, 논문)을 찾아 교체. 시간 많이 듦.

### 옵션 3 — G7 내부설치 확장 (사용자 요청)
BUILDING_OFFSET 제약을 "외부설치 OR 내부설치(클라이밍)" 둘 다 허용하도록 수정. 역삼동 feasible 해결.

**반드시 먼저 고쳐야 할 오류 (논문 신뢰성 직결):**
- WIND_SPEED 35→20 (또는 생존풍속과 작업풍속 구분 명확화)
- MR 160C 반경 50→61.1 (또는 의도적 제한이면 명시)
- 크레인 사양 2중정의 통합
- handoff vs 코드 자립고 불일치(67 vs 30) 확정
